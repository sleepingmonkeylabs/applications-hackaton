Architecture review: Grant application assistant
Your instinct is correct. Here's how to tighten each stage.

Stage 1 — Scrape → Embed → Index
What you have right: The cadence-based refresh pattern is correct for grants. Weekly or bi-weekly is typical.
What needs to be explicit:
Grants have two fundamentally different data types that need separate storage:
Data typeWhat it isStorageStructured metadataDeadline, max amount, eligibility criteria (org type, size, geography), application URL, method (email / web form)Relational table or JSON document storeSemantic contentFull criteria text, purpose, scoring rubricVector store (embedded chunks)
Don't embed the deadline or the application URL. They belong in metadata, filterable before retrieval. Embedding them just dilutes your vectors.
Chunking strategy matters here. Grants are section-structured documents (Purpose / Eligibility / Criteria / How to apply). Chunk at section boundaries, not at token count. A grant's eligibility section and its scoring rubric are semantically very different — they should be separate vectors with a grant_id foreign key so you can reassemble the full grant if needed.

Stage 2 — Query → Retrieve → Pull rules
This is actually a two-stage retrieval, which is the right pattern. Name it clearly:
Stage 2a:  Semantic search → top-K candidate grant_ids  (vector similarity)
Stage 2b:  Metadata fetch → full structured record for those grant_ids  (SQL/JSON lookup)
"Pull rules" = Stage 2b. The "rules" are the structured eligibility constraints + scoring criteria. These are what constrain your generation prompt.
Critical gap: pre-filtering before vector search. Before hitting the vector store, filter by:

deadline > today — this is the highest-stakes filter. Generating a draft for an expired grant is a hard failure.
Eligibility match (org type, geography) if you can derive those from the user's question

Without deadline filtering, you will retrieve expired grants. Embeddings don't know about time.

Stage 3 — Generate 3 ansökan drafts
What you have right: Three drafts is a good pattern — gives the user optionality without overwhelming them.
Prompt structure to use:
System:  You are a grant application writer for [org context].
         Apply ONLY to the criteria listed below. Do not invent requirements.
         
Grant:   [full structured record from Stage 2b]
Criteria: [chunked criteria text from Stage 2a]
User question: [original query]

Generate an application draft that addresses each scoring criterion explicitly.
Tag each paragraph with the criterion it addresses.
The tagging instruction is important — it makes hallucination auditable. If a paragraph can't be tagged to a criterion, it's probably confabulated.
Differentiate the three drafts on a strategic axis, not randomly:

Draft A: emphasizes impact/outcomes
Draft B: emphasizes method/execution rigor
Draft C: emphasizes team/organizational credibility


Stage 4 — Application method suggestions
This is not a RAG problem. It's a structured metadata lookup. The application method (email address, online form URL, portal login, PDF to post) is scraped once and stored as a structured field on the grant record.
Your output package per grant should be:
{
  grant_name: ...,
  deadline: ...,
  max_amount: ...,
  draft: "...",
  application_method: {
    type: "online_form" | "email" | "pdf_post",
    url: "...",
    contact_email: "...",
    notes: "requires BankID login" 
  }
}
The "suggestions on how to apply" is just rendering this structured block, not generation.
